PHASE 3 · WEB PATH · PICK

FIFO 先进先出
校验算法

原则：同一物料，先入库的批次必须先被选料出库。不允许跳过更早的批次去取较晚批次。

FIFO 校验算法（按物料分组）

1取选中批次中最晚的 CreateTimeSelectedLatestDate
2查同物料未选批中最晚的 CreateTimeUnselectedEarliestDate
3比较: SelectedLatestDate > UnselectedEarliestDate ?按天截断对比
✓
选中最晚 <= 未选最早 → 通过
✗
选中最晚 > 未选最早 → "批次{code}未遵守先进先出原则"

筛选条件: IsDelete==0, ProductionStatus==0, RemainQty>0, Warehouse in [01,42], ExpiredDatetime > Now

PHASE 3 · WEB PATH · OUTSAVE

扫码 OutSave: 9 道系统校验关卡

API: POST /api/wms/inout/outSave — 每扫一个条码都走完整 9 道校验

1
条码首字符 ≠ 'D' — D开头=不良品，拒绝出库
2
批次在 WareHouseStock 存在
3
ProductionStatus ≠ 2 (Send) — 已出库不可重复扫
4
ExpiredDatetime >= 当前时间 — 过期拒绝(XSOUT/FGOUT豁免)
5
QCStatus ≠ 1 — 质量锁定拒绝(返工FGOUT豁免)
6
QCSnHoldDetail.ProductStatus ≠ Hold — SN级锁定
7
客户料号与出库单明细 匹配(XSOUT Only)
8
客户周标签与包装周标签 一致(IsCheckWeek=Y时)
9
扫码数量 ≤ 明细 Qty - ActQty — 不超需发量

PHASE 3 · WEB PATH · OUTSAVE

质量锁定拦截
两层检查机制

核心原则：被质量锁定的批次和 SN 码不得出库，确保不合格品不流向客户。

第一层: 批次级 QCStatus

WareHouseStock.QCStatus == 1

整批次被质量部门标记为锁定
→ 拒绝出库: "批次已被质量锁定"

例外: FGOUT(返工发料)不检查此项

第二层: SN 级 QCSnHoldDetail

QCSnHoldDetail.ProductStatus == Hold

精确到单个 SN 的质量锁定
PackNo = 条码号 → 查到即锁定
→ 拒绝出库: "批次已被质量锁定"

例外: FGOUT(返工发料)不检查此项

PHASE 3 · WEB PATH · OUTSAVE

客户料号校验
& 周标签校验

XSOUT 销售出库专属的两项额外校验，确保发给客户的物料和标识完全正确。

客户料号校验 (IsCheckCustomerMaterial=Y)

对比 IronCorePackaging 中的客户料号 vs 出库单明细中的 CustomerMaterialCode

匹配 → AgreeQty 统计+1
不匹配 → DifQty 统计+1 (差异记录)

周标签校验 (IsCheckWeek=Y)

提取包装标签(IronCorePackaging.CreateTime)的 ISO 周号
对比客户二维码(CustomerQRCode)第5段的周号

不一致 → "客户标签日期与包装标签日期不一致"

PHASE 4 · LOGGING

接口日志 & ErpStatus 状态机

ERP_MESInterFaceLog 日志表

每次 MOM → ERP 调用都写入此表：
· 请求 JSON (截断 60,000 字节)
· 响应 JSON (截断 60,000 字节)
· 调用时间、表单 ID、操作类型
· 成功/失败状态

附加: ERP_MESInterFaceLog_File 备用日志表

ErpStatus 状态机

0 → 待处理 Submit后初始状态，后台Job检测
1 → 成功 ERP返回单据号，写入ErpOrder字段
2 → 失败 ERP返回错误，ErpMsg记录错误信息

失败恢复: POST /api/wms/kt/rePushXSOutToErp
将 ErpStatus=2 重置为 0 → 触发重推
失败 Rollback 机制： ERP 保存失败时，执行 UnAudit + Delete 回滚 ERP 侧已创建（但未审核）的单据，确保两端数据一致。

EXCEPTION HANDLING

异常处理: DeleteOut 扫码回退

当操作员扫错条码或需要撤销时，调用 DeleteOut 反向恢复。

1校验: Order 不能已是 Outbounded"该出库单已出库完成，不可删除"
2Detail.ActQty -= receive.Qty扣回已发量
3Stock: ProductionStatus → Create(0)RemainQty=SendQty, SendQty=0
4FHTZ Source: receive 软删除Source Detail/Order 状态回退
5Receive: pick-mode → WaitFinishdirect-scan → IsDelete=1
6Order Status: Outbounding → WaitOutboundDetail Status 对应回退

EXCEPTION HANDLING

ERP 推送失败
重推机制

当 MOM → ERP 回写失败时，系统提供手动和自动两种恢复方式，确保数据最终一致性。

手动重推

API: POST /api/wms/kt/rePushXSOutToErp

1. 查询 ErpStatus=2 的单据
2. 将 ErpStatus 重置为 0
3. 后台 Job 检测到 0 → 重新调用 SAL_OUTSTOCKSave()
4. 成功 → 1, 失败 → 2(保留错误信息)

自动重试

后台定时任务周期性扫描：

· 扫描 ErpStatus = 0 的待推送单据
· 调用对应 ERP Service 推送
· 成功标记 ErpStatus=1

同一单据最多重试 N 次
超过阈值 → 人工介入
